# GitHub Star Weekly 项目总控计划

更新时间：2026-05-21

## 1. 项目目标

做一个每周一自动运行的工具：

- 统计全 GitHub 上一个完整统计周期内新增 Star 最多的 10 个项目。
- 给每个项目生成中文介绍。
- 标注每个项目历史累计上榜次数。
- 生成 HTML 邮件并发送到指定邮箱。
- 保存历史榜单，确保同一周重复运行不会重复发邮件。

第一版只追求稳定可用，不做网页、用户系统、多订阅者和复杂图表。

## 3. 推荐工程方案

使用 GitHub Actions 每周定时执行 Python 程序。

数据流如下：

```text
GitHub Actions 定时触发
  -> Python 主程序启动
  -> BigQuery 查询 GH Archive 的 WatchEvent
  -> 得到本周新增 Star Top 10
  -> GitHub API 补充项目详情和 README
  -> OpenAI API 生成中文介绍
  -> SQLite 保存榜单历史
  -> Jinja2 渲染 HTML 邮件
  -> Resend 发送邮件
  -> SQLite 记录已发送
```

## 4. 为什么这样设计

### 4.1 为什么用 GH Archive + BigQuery

GitHub 官方没有直接提供“全站过去 7 天新增 Star 排行榜”接口。GH Archive 会记录公开 GitHub 事件，其中 `WatchEvent` 基本代表用户新增 Star。BigQuery 适合在大量历史事件里做统计，所以我们用 SQL 直接按仓库分组计数。

### 4.2 为什么不用 GitHub Trending

GitHub Trending 是页面产品，不是稳定 API，而且它的算法不等于“新增 Star 数”。我们要的是可重复、可解释、可验证的统计口径，所以使用事件计数。

### 4.3 为什么要 SQLite

这个工具每周只写 10 条榜单数据，数据库规模很小。SQLite 是一个单文件数据库，简单、便宜、容易随仓库保存。它还能帮我们判断某个项目历史上榜几次，以及某周邮件是否已经发过。

### 4.4 为什么要幂等

幂等的意思是：同一件事运行多次，结果不会乱。比如同一周程序失败后重跑，不应该发两封一样的邮件。我们用数据库里的 `email_sends.week_start` 防止重复发送。

### 4.5 为什么要 fallback

外部服务都有可能失败：GitHub API 限流、OpenAI API 超时、README 不存在、邮件服务临时失败。稳定工具不能因为一个项目摘要失败就整封邮件失败，所以每个环节都要有降级方案。

## 5. 统计口径

第一版建议使用“上一个完整 UTC 自然周”：

- 每周一中国时间 09:00 左右运行。
- GitHub Actions cron 对应 UTC 时间可用 `0 1 * * 1`。
- 统计上周一至上周日的 GH Archive 日表。
- 指标是 `WatchEvent` 数量，代表新增 Star 事件数。
- 这是新增 Star 事件数，不是净增长；取消 Star 不会扣除。

如果后续要严格按中国时间自然周统计，需要跨 UTC 日期边界过滤时间戳，查询会稍微复杂一些。第一版优先稳定和低成本。

## 6. 并行线程分工

你可以开 5 个线程并行推进。每个线程只负责自己的边界，避免互相改同一批文件。

### 线程 A：项目骨架和配置

目标：把空仓库变成可运行的 Python 项目骨架。

负责文件：

- `requirements.txt`
- `.gitignore`
- `README.md`
- `src/config.py`
- `src/__init__.py`
- `data/.gitkeep`

交付标准：

- 本地能安装依赖。
- 配置全部从环境变量读取。
- README 说明本地运行、Secrets 配置、统计口径。

### 线程 B：数据采集

目标：从 BigQuery 查询 Top 10，并用 GitHub API 补充仓库信息。

负责文件：

- `src/collect.py`
- `src/enrich.py`
- 相关测试文件

交付标准：

- BigQuery 查询限制在 7 天内。
- 使用参数化 SQL。
- GitHub API 单个仓库失败不会中断全部流程。
- README 不存在时能继续。

### 线程 C：数据库和幂等

目标：保存历史榜单、计算历史上榜次数、防止重复发送。

负责文件：

- `src/db.py`
- `data/rankings.sqlite` 的初始化逻辑
- 相关测试文件

交付标准：

- 自动建表。
- 支持榜单 upsert。
- 能计算包含本周的累计上榜次数。
- 能检查某周是否已发送。

### 线程 D：摘要、邮件模板、发送

目标：生成中文介绍，渲染 HTML，调用 Resend 发邮件。

负责文件：

- `src/summarize.py`
- `src/render.py`
- `src/emailer.py`
- `templates/weekly_email.html`
- 相关测试文件

交付标准：

- OpenAI 失败时有 fallback 中文介绍。
- 邮件包含 Top 10 表格和项目详情。
- Resend 发送失败时不写入已发送记录。
- HTML 简洁，邮箱兼容优先。

### 线程 E：主流程和 GitHub Actions

目标：把所有模块串起来，并让 GitHub Actions 每周自动运行。

负责文件：

- `src/main.py`
- `.github/workflows/weekly.yml`
- 端到端 smoke test

交付标准：

- 支持手动运行 `python -m src.main`。
- 支持 `--force` 测试重发。
- workflow 支持定时和手动触发。
- workflow 运行后提交更新后的 SQLite 数据库。

## 7. 线程协作规则

- 每个线程先看本总控计划和 `流程表.md`。
- 每个线程只改自己负责的文件。
- 任何密钥都不能写进代码、README 示例或测试快照。
- 所有外部 API 都要有超时、错误信息和降级逻辑。
- 写完后说明：改了哪些文件、如何验证、还有什么风险。

## 8. 开发顺序

虽然可以并行，但推荐集成顺序如下：

1. 先合并线程 A，得到项目骨架。
2. 合并线程 C，先有数据库能力。
3. 合并线程 B，跑通 Top 10 数据。
4. 合并线程 D，生成邮件内容。
5. 合并线程 E，串起完整流程和自动化。
6. 本地跑一次完整流程。
7. GitHub Actions 手动触发一次。
8. 等下一个周一自动运行。

## 9. 客户需要准备的账号和密钥

需要你自己注册或创建：

- Google Cloud 项目和 BigQuery Service Account JSON。
- GitHub Token，用于读取公开仓库信息。
- OpenAI API Key，用于生成中文介绍。
- Resend API Key，用于发送邮件。
- 发件邮箱和收件邮箱。

这些密钥最终放到 GitHub 仓库的 Actions Secrets 里，不提交到代码。

## 10. 第一版验收标准

满足下面条件就算第一版完成：

- 手动运行能收到邮件。
- 邮件中有 Top 10 项目。
- 每个项目有新增 Star、总 Star、语言、中文介绍、历史上榜次数。
- 同一周重复运行默认不会重复发邮件。
- GitHub Actions 可手动触发成功。
- GitHub Actions 能按周一定时运行。
- SQLite 中能保留历史榜单。

## 11. 关键风险和对策

| 风险 | 对策 |
|---|---|
| BigQuery 扫描太多导致费用 | 固定 7 天范围，dry run 记录扫描量 |
| GitHub API 限流 | 使用 token，单仓库失败不中断 |
| AI 摘要编造 | 只给真实字段和 README 片段，提示词要求信息不足就明说 |
| 邮件重复发送 | `email_sends` 表 + Resend 幂等键 |
| 邮件进垃圾箱 | 优先验证自有域名，HTML 保持简单 |
| Actions 定时失败没人发现 | 保留 workflow 日志，后续可加失败通知 |
