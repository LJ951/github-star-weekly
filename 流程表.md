# GitHub 每周 Star 增长榜工具流程表

目标：做一个每周一自动统计“全 GitHub 过去 7 天新增 Star 最多的 10 个项目”，生成中文介绍，标注每个项目历史上榜次数，并发送到邮箱的稳定工具。

推荐技术栈：

- Python：主程序
- GitHub Actions：每周定时运行
- GH Archive + Google BigQuery：统计全 GitHub 新增 Star 事件
- GitHub REST API：补充仓库信息
- SQLite：保存历史榜单和发送记录
- OpenAI API：生成中文项目介绍
- Resend：发送邮件

---

## 1. 准备账号和密钥

### 1.1 准备 GitHub 账号

用途：

- 存放项目代码
- 运行 GitHub Actions 定时任务
- 保存 SQLite 数据库文件，或至少保存程序日志

操作：

1. 登录 GitHub。
2. 新建一个私有仓库，例如：`github-star-weekly`。
3. 后续所有代码都放进这个仓库。

### 1.2 准备 Google Cloud / BigQuery

用途：查询 GH Archive 数据集中的 GitHub 公共事件。

操作：

1. 打开 Google Cloud Console。
2. 新建一个 Project，例如：`github-star-weekly`。
3. 启用 BigQuery API。
4. 确认可以访问公开数据集：`githubarchive`。
5. 创建一个 Service Account。
6. 给 Service Account 授权：
   - `BigQuery Job User`
   - `BigQuery Data Viewer`
7. 创建 JSON 格式的 Service Account Key。
8. 保存 JSON 内容，后续放入 GitHub Secrets。

注意：

- 每周只查询一次最近 7 天数据，正常个人使用大概率在免费额度内。
- 查询时必须限制日期范围，避免扫描过多数据。

### 1.3 准备 GitHub Token

用途：调用 GitHub API 获取仓库详情。

操作：

1. 打开 GitHub Settings。
2. 进入 Developer settings。
3. 创建 Fine-grained personal access token 或 classic token。
4. 只需要公开仓库读取权限。
5. 保存 token。

### 1.4 准备 OpenAI API Key

用途：生成中文项目介绍。

操作：

1. 打开 OpenAI API 平台。
2. 创建 API key。
3. 保存 key。

建议：

- 第一版使用低成本模型。
- 只给 Top 10 项目生成介绍，不要把全量 README 都传给模型。

### 1.5 准备 Resend API Key

用途：发送邮件。

操作：

1. 注册 Resend。
2. 验证发送域名，或先用 Resend 提供的测试发件地址。
3. 创建 API key。
4. 准备收件邮箱。

## 4. 设计数据库

使用 SQLite。第一版只需要两个核心表。

### 4.1 榜单表

```sql
CREATE TABLE IF NOT EXISTS weekly_rankings (
  week_start TEXT NOT NULL,
  week_end TEXT NOT NULL,
  rank INTEGER NOT NULL,
  full_name TEXT NOT NULL,
  html_url TEXT NOT NULL,
  stars_gained INTEGER NOT NULL,
  total_stars INTEGER,
  language TEXT,
  description TEXT,
  topics TEXT,
  summary_zh TEXT,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (week_start, full_name)
);
```

### 4.2 邮件发送记录表

```sql
CREATE TABLE IF NOT EXISTS email_sends (
  week_start TEXT PRIMARY KEY,
  week_end TEXT NOT NULL,
  sent_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  recipient TEXT NOT NULL,
  message_id TEXT
);
```

用途：

- `weekly_rankings` 用来保存每周 Top 10。
- `email_sends` 用来防止同一周重复发邮件。

## 5. 固定统计口径

建议口径：

- 每周一上午 9 点，中国时间发送邮件。
- 统计窗口：上周一 00:00:00 到上周日 23:59:59，使用 UTC 日期表查询。
- 指标：GH Archive 中 `WatchEvent` 数量。
- 排名：按 `WatchEvent` 数量从高到低。

注意：

- `WatchEvent` 基本代表新增 Star。
- 这不是净增长，取消 Star 不会扣除。
- 这个口径比 GitHub Trending 更接近“全 GitHub 新增 Star 最多”。

## 6. 查询 GH Archive

`collect.py` 的核心逻辑：

1. 根据当前日期计算上周一和上周日。
2. 生成 BigQuery `_TABLE_SUFFIX` 范围。
3. 查询 `githubarchive.day.*`。
4. 过滤 `type = 'WatchEvent'`。
5. 按 `repo.name` 分组统计。
6. 返回 Top 10。

示例 SQL：

```sql
SELECT
  repo.name AS full_name,
  COUNT(*) AS stars_gained
FROM `githubarchive.day.*`
WHERE
  _TABLE_SUFFIX BETWEEN @start_suffix AND @end_suffix
  AND type = 'WatchEvent'
GROUP BY full_name
ORDER BY stars_gained DESC
LIMIT 10;
```

稳定性要求：

- 使用参数化查询。
- 查询前做 dry run，记录预计扫描量。
- 查询失败时抛出明确错误。
- 不要查询超过 7 天的数据。

## 7. 补充 GitHub 仓库信息

`enrich.py` 的核心逻辑：

对 Top 10 每个仓库调用：

```text
GET https://api.github.com/repos/{owner}/{repo}
```

需要保存字段：

- `full_name`
- `html_url`
- `description`
- `stargazers_count`
- `language`
- `topics`
- `created_at`
- `updated_at`
- `pushed_at`

可选再调用 README：

```text
GET https://api.github.com/repos/{owner}/{repo}/readme
```

注意：

- README 可能不存在，不能因为一个项目失败导致整个流程失败。
- README 内容要截断，例如最多取前 6000 字符。
- GitHub API 要加 token，避免限流太低。

## 8. 计算历史上榜次数

在保存本周榜单之前或之后，都可以查询历史次数。

推荐逻辑：

```sql
SELECT COUNT(*)
FROM weekly_rankings
WHERE full_name = ?
  AND rank <= 10;
```

展示时可以这样写：

- 首次上榜：`首次上榜`
- 非首次：`历史第 3 次上榜`

注意：

- 如果先保存本周榜单再统计，次数包含本周。
- 如果先统计再保存，次数不包含本周。
- 建议邮件中展示“包含本周的累计上榜次数”，更直观。

## 9. 生成中文项目介绍

`summarize.py` 的输入：

- 仓库名
- GitHub description
- topics
- language
- README 前几千字符
- 本周新增 Star
- 当前总 Star

建议输出格式：

```text
这个项目是什么：1-2 句话
为什么本周增长快：1-2 句话
适合谁关注：1 句话
风险或注意点：1 句话
```

提示词建议：

```text
你是一名技术趋势分析师。请根据下面的 GitHub 仓库信息，用中文介绍这个项目。
要求：
1. 不要夸大，不要编造 README 中没有的信息。
2. 说明项目解决什么问题。
3. 说明可能为什么本周增长快。
4. 说明适合哪些开发者关注。
5. 如果信息不足，请明确说“项目信息有限”。
6. 控制在 180-250 字。
```

稳定性要求：

- LLM 调用失败时，使用 fallback 摘要。
- fallback 可以基于 description、language、topics 拼一段简短介绍。
- 不要因为一个项目摘要失败导致邮件不发送。

## 10. 渲染 HTML 邮件

邮件应包含：

1. 标题：`GitHub 本周 Star 增长最快 Top 10`
2. 统计周期：例如 `2026-05-11 至 2026-05-17`
3. 说明口径：`基于 GH Archive WatchEvent 统计，代表新增 Star 事件数`
4. Top 10 表格：
   - 排名
   - 项目
   - 本周新增 Star
   - 当前总 Star
   - 语言
   - 历史上榜次数
5. 每个项目的详细中文介绍
6. 页脚：数据源、生成时间、退订说明

邮件模板建议保持简单，避免复杂 CSS。

---

## 11. 发送邮件

`emailer.py` 使用 Resend。

发送前检查：

1. 当前 `week_start` 是否已经在 `email_sends` 中存在。
2. 如果已经存在，默认不重复发送。
3. 如果需要测试，可以提供 `--force` 参数。

发送成功后保存：

- `week_start`
- `week_end`
- `recipient`
- `message_id`
- `sent_at`

失败处理：

- 邮件发送失败时，不写入 `email_sends`。
- 这样下一次重跑可以继续发送。

## 12. 主流程 main.py

推荐执行顺序：

```text
1. 读取配置
2. 初始化数据库
3. 计算统计周期
4. 检查本周邮件是否已发送
5. 查询 BigQuery 获取 Top 10
6. 调 GitHub API 补充仓库信息
7. 调 OpenAI 生成中文介绍
8. 写入 weekly_rankings
9. 查询每个项目累计上榜次数
10. 渲染 HTML 邮件
11. 发送邮件
12. 写入 email_sends
13. 输出日志
```

幂等规则：

- `weekly_rankings` 使用 `PRIMARY KEY (week_start, full_name)`。
- 重跑同一周时使用 upsert。
- `email_sends` 用 `week_start` 做主键。
- 默认不重复发邮件。

## 13. GitHub Actions 定时运行

`.github/workflows/weekly.yml`：

```yaml
name: Weekly GitHub Star Report

on:
  schedule:
    - cron: "0 1 * * 1"
  workflow_dispatch:

jobs:
  weekly:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run weekly report
        env:
          GOOGLE_APPLICATION_CREDENTIALS_JSON: ${{ secrets.GOOGLE_APPLICATION_CREDENTIALS_JSON }}
          GITHUB_TOKEN: ${{ secrets.GH_TOKEN }}
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          RESEND_API_KEY: ${{ secrets.RESEND_API_KEY }}
          EMAIL_FROM: ${{ secrets.EMAIL_FROM }}
          EMAIL_TO: ${{ secrets.EMAIL_TO }}
        run: python -m src.main

      - name: Commit database changes
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add data/rankings.sqlite
          git commit -m "Update weekly ranking database" || echo "No changes to commit"
          git push
```

说明：

- `cron: "0 1 * * 1"` 表示 UTC 周一 01:00，即中国时间周一 09:00。
- 如果数据库改用 Supabase，则不需要 commit SQLite 文件。
- `GOOGLE_APPLICATION_CREDENTIALS_JSON` 需要程序运行时写入临时 JSON 文件供 BigQuery 客户端使用。

## 14. 配置 GitHub Secrets

在 GitHub 仓库中进入：

```text
Settings -> Secrets and variables -> Actions -> New repository secret
```

添加：

| Secret 名称 | 内容 |
|---|---|
| `GOOGLE_APPLICATION_CREDENTIALS_JSON` | Google Service Account JSON 全文 |
| `GH_TOKEN` | GitHub API Token |
| `OPENAI_API_KEY` | OpenAI API Key |
| `RESEND_API_KEY` | Resend API Key |
| `EMAIL_FROM` | 发件邮箱 |
| `EMAIL_TO` | 收件邮箱 |

注意：

- 不要把任何 key 写进代码。
- 不要把 `.env` 提交到 GitHub。

## 19. 第一版验收标准

当下面条件都满足时，第一版就算完成：

- 能手动运行并成功收到邮件。
- 邮件包含 Top 10 项目。
- 每个项目有本周新增 Star、当前总 Star、语言、中文介绍。
- 每个项目显示累计上榜次数。
- 同一周重复运行不会重复发邮件。
- GitHub Actions 能在周一自动运行。
- 数据库中能看到历史榜单记录。

## 20. 关键风险

| 风险 | 处理方式 |
|---|---|
| BigQuery 扫描过多导致费用增加 | 限制 7 天日期表，使用 dry run |
| GitHub API 限流 | 使用 token，缓存元数据 |
| LLM 生成内容不准确 | 限制输入来源，提示词要求不编造 |
| 邮件进垃圾箱 | 使用验证域名，减少营销化文案 |
| 重复发送邮件 | 使用 `email_sends` 表做幂等 |
| GitHub Actions 失败无人发现 | 配置失败通知，定期看 Actions 日志 |

---

## 21. 最小可用版本范围

第一版只做这些：

- 全 GitHub 周新增 Star Top 10
- 中文介绍
- 历史上榜次数
- HTML 邮件
- 每周一自动发送
- SQLite 历史数据

第一版暂时不做这些：

- Web 页面
- 用户系统
- 多人订阅
- 退订管理
- 付费系统
- 复杂图表

这样可以把开发周期控制在 1-3 天内，后续再逐步增强。
